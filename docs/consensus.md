# Consensus — Simplex BFT

## Algorithm

Simplex is a 3-phase BFT consensus protocol from Commonware. It achieves ~600 ms finality with 4 validators and 10 ms simulated network latency.

### Phases per View

```
View N
 │
 ├─ Propose   ── elected leader broadcasts a block proposal
 │
 ├─ Notarize  ── validators vote on the proposal
 │               Q-quorum (≥80% stake) → Notarization certificate issued
 │
 └─ Finalize  ── validators finalize the notarized block
                 Q-quorum finalize votes → Finalization certificate issued
```

### Quorums

| Quorum | Threshold | Purpose |
|---|---|---|
| L-quorum | ≥40% | Advance to next view (liveness — even if no block finalized) |
| Q-quorum | ≥80% | Notarization / Finalization (safety) |

If a leader is unresponsive, validators broadcast a **nullify** vote. Once L-quorum of nullifies is reached, the view is **nullified** and consensus advances to view N+1.

### Safety & Liveness

- **Safety**: requires ≤20% Byzantine validators (Q-quorum threshold)
- **Liveness**: requires ≥60% honest validators online (L-quorum threshold)
- With 4 validators, 0 Byzantine: all properties hold with wide margin

### Leader Election

`RoundRobin<Sha256>` — deterministic round-robin based on view number, using a SHA-256 hash of the current certificate to break ties.

## Commonware Simplex API

### Core Types

```rust
// Consensus engine — 5 generic parameters after the context
Engine<deterministic::Context, ed25519::Scheme, RoundRobin, Sha256Digest, Sequential>

// Config struct (all fields required)
config::Config {
    blocker,             // Oracle::control(validator) — network flow control
    scheme,              // ed25519::Scheme — signing & cert verification
    elector,             // RoundRobin — leader election
    automaton,           // impl Automaton — block proposal & verification
    relay,               // impl Relay — broadcast finalized blocks to app
    reporter,            // impl Reporter — observe all consensus activity
    partition,           // String — storage namespace (unique per validator)
    mailbox_size: 1024,
    epoch: Epoch::new(0),
    leader_timeout:        Duration::from_secs(1),
    certification_timeout: Duration::from_secs(2),
    timeout_retry:         Duration::from_secs(10),
    fetch_timeout:         Duration::from_secs(1),
    activity_timeout:  Delta::new(10),   // views before marking validator inactive
    skip_timeout:      Delta::new(5),    // views before allowing view skip
    fetch_concurrent: 1,
    replay_buffer:  NZUsize!(1024 * 1024),
    write_buffer:   NZUsize!(1024 * 1024),
    page_cache:     CacheRef::from_pooler(&ctx, NZU16!(1024), NZUsize!(10)),
    strategy: Sequential,
}
```

### Starting an Engine

```rust
let engine = Engine::new(ctx.with_label("engine"), engine_cfg);

// Three p2p channel pairs:
//   pending   = channel 0 — vote messages
//   recovered = channel 1 — certificate messages
//   resolver  = channel 2 — block fetch / resolver messages
engine.start(pending, recovered, resolver);
```

### Observing Finalization

```rust
// reporter::Reporter implements Monitor<Index = View>
let (mut latest, mut monitor): (View, Receiver<View>) = reporter.subscribe().await;
while latest.get() < target {
    latest = monitor.recv().await.unwrap();
    println!("finalized view {}", latest.get());
}
```

The `Reporter` mock also exposes internal state for inspection:
```rust
reporter.finalizations.lock()   // HashMap<View, Finalization<...>>
reporter.notarizations.lock()   // HashMap<View, Notarization<...>>
reporter.faults.lock()          // detected Byzantine evidence
reporter.invalid.lock()         // count of invalid messages received
```

## Ed25519 Certificate Scheme

Validators use Ed25519 for both identity and signing. The scheme is generated by the `impl_certificate_ed25519!` macro in `commonware-cryptography`.

```rust
// Generate N validator key sets for testing
let fixture = ed25519::fixture(&mut context, namespace, n);
// fixture.participants — Vec<Ed25519PublicKey>, sorted deterministically
// fixture.schemes      — Vec<ed25519::Scheme>, one per validator (includes private key)
// fixture.verifier     — ed25519::Scheme (verify-only, no private key)
```
